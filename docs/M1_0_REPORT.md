# Milestone 1.0 — report (Windows native build bring-up + relay)

**Status: complete and verified on both OSes.** The existing M0 `offscreen_png`
example now builds and runs **natively on Windows** with the MSVC-family
toolchain (clang-cl + MSBuild) and renders on the **native NVIDIA RTX 4090**
Vulkan driver, with **no patches** to the vendored rive-runtime. The Linux build
is unchanged and regression-free. A reusable relay (`scripts/win.cmd`) lets WSL2
drive the native Windows build.

rive's Vulkan SPIR-V is generated by the **Vulkan SDK** on a clean checkout / CI
(the hermetic path); a working tree carrying prebuilt SPIR-V (e.g. synced from a
Linux build) skips that step locally. So "no Vulkan SDK" is a *local* convenience,
not a build property — see §2.

```
# from WSL2 (repo synced to E:\DEV\rive-rust):
cmd.exe /c "scripts\win.cmd run --release --example offscreen_png -- assets\coffee_loader.riv out_win.png"

# or from a native Windows terminal, at E:\DEV\rive-rust:
scripts\win.cmd run --release --example offscreen_png -- assets\coffee_loader.riv out_win.png
```

No Bevy, no plugin, no wgpu/shared-device code was added — this stage is build
system only, exactly as M0 was for Linux. The C ABI and the safe wrapper are
untouched.

---

## 1. What built and what runs (Windows)

- **rive-runtime PLS Vulkan static libs**, built from the pinned submodule via
  `premake5 vs2022` → **MSBuild** with the **ClangCL** platform toolset. All ten
  produced (flat in `out/rive-rust-m0/`, `.lib`, **no** `lib` prefix):
  `rive.lib`, `rive_pls_renderer.lib`, `rive_decoders.lib`, `libpng.lib`,
  `zlib.lib`, `libjpeg.lib`, `libwebp.lib`, `rive_harfbuzz.lib`,
  `rive_sheenbidi.lib`, `rive_yoga.lib`.
- **C++ shim** + rive's `rive_vk_bootstrap` sources, compiled with **clang-cl**
  via the `cc` crate (to match the libs' ABI), `/MT` static CRT.
- **`offscreen_png.exe`** links (static CRT) and runs end-to-end on two samples,
  output **byte-comparable to the Linux M0 reference**:
  - `coffee_loader.riv` → `out_win.png` — **30,105** / 262,144 non-background
    px (Linux M0: 30,104). Upright, centered coffee mug.
  - `octopus_loop.riv` → `out_win_octopus.png` — **75,985** px. Upright (blue hat
    on top, tentacles down) — the orientation flip-fix from M0 carries over.

### Device actually used

Native Windows exposes exactly one Vulkan ICD — the RTX 4090
(`vulkaninfo`: **Vulkan 1.4.321**, driver 591.86, device
`NVIDIA GeForce RTX 4090`, **`VK_EXT_fragment_shader_interlock` present**). There
is no software rasterizer (llvmpipe) installed, so the render necessarily ran on
the 4090. Unlike WSL2 — where the only ICD is Mesa **Dozen** (Vulkan-on-D3D12,
atomic-only) — the native driver advertises fragment-shader-interlock, so the
Windows build exercises rive's **interlock PLS fast path**. (rive's bootstrap
init message routes to `OutputDebugString` on Windows rather than the console, so
`RIVE_DEBUG=1` doesn't echo the device to stdout here; the device is confirmed via
`vulkaninfo` + the fact that `vulkan-1.dll` is the NVIDIA driver's loader.)

## 2. Shader provenance & the no-submodule-patch property

rive's Vulkan SPIR-V is produced by `glslangValidator`/`spirv-opt` (the Vulkan
SDK), invoked only in rive's configure-time shader step. There are two supported
ways to satisfy it; `build.rs` requires exactly one and **fails early with the
remedy** if neither is present:

1. **Hermetic path (CI / clean checkout): the Vulkan SDK.** With the SDK installed
   (`VULKAN_SDK` set; the relay adds its `Bin` to PATH), rive's shader step
   generates the SPIR-V natively. This is the canonical Windows build and has **no
   Linux dependency** — it's what `.github/workflows/windows.yml` does.
2. **Local fast path: reuse prebuilt SPIR-V.** If the tree already contains the
   116 OS-independent SPIR-V headers under
   `renderer/out/rive-rust-m0/include/generated/shaders/spirv/` (e.g. synced from
   a Linux build via `scripts/sync_to_windows.sh`), rive's `make` step reports
   `Nothing to be done for 'spirv'` and the SDK is not invoked. A convenience, not
   a hermeticity guarantee.

> Earlier this reuse was the *only* mechanism, which made "no Vulkan SDK on
> Windows" an **incidental** property — a clean Windows checkout (CI, a fresh box,
> a teammate) would have failed at rive's shader step for want of the SDK. The
> Vulkan SDK is now the intentional CI path, and `build.rs` fails fast with an
> actionable message ("install the LunarG Vulkan SDK, or build from a tree with
> prebuilt SPIR-V") if neither is available — no more cryptic
> `glslangValidator: not found` from deep inside `make`.

Two things still need **no** SDK and **no** patch on Windows:

- **D3D shaders** rive always builds on a Windows target use the Windows SDK's
  `fxc` (present via VS), not the Vulkan SDK.
- **Vulkan is loaded at runtime** — rive is built `VK_NO_PROTOTYPES` + VMA dynamic
  functions; `rive_vk_bootstrap` does `LoadLibraryA("vulkan-1.dll")`, so there is
  **no Vulkan import lib to link** (`build.rs` emits no Vulkan link directive on
  Windows); the NVIDIA driver's `vulkan-1.dll` satisfies it at runtime.

This matches how rive-runtime is built on Windows (its `build_rive.sh` runs under
Git Bash/MSYS with `make` + premake `vs2022`→MSBuild + clang-cl + the Vulkan SDK)
— so no source or build-file patch is required.

## 3. Exact Windows toolchain used

| Tool | Version / source |
| --- | --- |
| rustc / cargo | 1.96.0 (`x86_64-pc-windows-msvc`); bumped from 1.93 to meet the 1.94 MSRV |
| Visual Studio | 2022 Community; **C++ Clang tools for Windows** component (clang-cl) |
| clang-cl | LLVM under `VC\Tools\Llvm\x64\bin` (rive's default toolset) |
| MSBuild | VS2022 (`-p:Platform=x64 -p:Configuration=default`) |
| fxc | Windows SDK 10.0.22621.0 (D3D shader compile) |
| premake5 | **5.0.0-beta7** (`tools/premake5.exe`); beta2 mistranslates rive's `fatalwarnings` for MSVC |
| GNU make | via Chocolatey (`C:\ProgramData\chocolatey\bin\make.exe`) |
| sh | Git for Windows (`C:\Program Files\Git\usr\bin\sh.exe`) — make's recipe shell |
| python | 3.11 (shader minify; not on the critical path because outputs are reused) |
| git | for the configure-time Vulkan-Headers/VMA/DirectX-Headers clones |
| Vulkan SDK | build-time on CI / clean checkouts (`glslangValidator` + `spirv-opt` → SPIR-V); skipped locally when prebuilt SPIR-V is synced (§2) |
| Vulkan runtime | `C:\Windows\System32\vulkan-1.dll` (NVIDIA driver) — loaded at runtime; no import lib |

rive libs are built `--config=debug`. On Windows rive forces
`staticruntime('on') + runtime('Release')` (= `/MT`) regardless of config, so the
debug libs use the **release static CRT**; Rust links `+crt-static`
(`.cargo/config.toml`) and `cc` builds the shim `/MT` to match — no
`libcmt`/`msvcrt` mismatch. Debug config also avoids release LTO bitcode (which a
non-LLVM final linker can't consume — see §3b).

## 3b. Performance caveat — rive built debug (resolve in M2, not M1a)

The rive libs are `--config=debug`: **optimization is off in the renderer hot
path.** The example links these debug libs into a `--release` binary, so **no
performance conclusions** are valid from M1.0 through M1a — any "how fast is the
native renderer?" impression is meaningless until rive is built optimized.

Why debug for now: rive's `--config=release` enables LTO that emits **LLVM
bitcode** archives, which Rust's default MSVC linker (`link.exe`) can't consume.
Two fixes, both **deferred to M2** (do not touch in M1a):

- **Preferred — `lld-link`.** The whole C++ side is already clang-cl/LLVM, so
  point Rust's MSVC linker at `lld-link` (one line in `.cargo/config.toml`:
  `[target.x86_64-pc-windows-msvc] linker = "lld-link"`); it consumes rive's
  release LTO bitcode, keeping LTO. Verify it doesn't fight Rust's link args.
- **Fallback — disable rive's LTO.** Build rive `--config=release --no-lto`
  (rive's `no-lto` option = `optimize On` without `LinkTimeOptimization`),
  yielding optimized non-bitcode `.lib`s the standard linker accepts.

`build.rs` already honors `RIVE_RUNTIME_CONFIG=release` to build release rive once
the linker question is settled.

## 4. Deliverables added in M1.0

- **`scripts/win.cmd`** — the relay. Locates VS via `vswhere`, sources
  `vcvars64.bat` (x64), prepends clang-cl and appends Git Bash `sh` + choco `make`
  to PATH (Git's `usr/bin` is appended, not prepended, so its coreutils `link.exe`
  doesn't shadow MSVC's `link.exe`), sets `WGPU_BACKEND=vulkan` (forward-looking,
  no effect yet), `cd`s to the repo root and forwards all args to `cargo`.
- **`crates/rive-renderer-sys/build.rs`** — now cross-platform: a Windows branch
  (premake `vs2022` → MSBuild/ClangCL, clang-cl shim, `.lib` discovery, runtime
  Vulkan / no import lib) alongside the unchanged Linux branch.
- **`.cargo/config.toml`** — `+crt-static` for `x86_64-pc-windows-msvc`.
- **`tools/fetch_premake.cmd`** — vendors premake **beta7** for Windows.
- **`scripts/sync_to_windows.sh`** — mirrors the working tree to `E:\DEV\rive-rust`
  for a Windows test (excludes Linux objects + `target/`; keeps vendor sources,
  the `.rive-deps` header cache, and the prebuilt SPIR-V headers).
- **`.gitattributes`** — LF for sources, CRLF for `.cmd`/`.bat`, binary assets
  untouched (the tree is edited from both WSL2 and Windows on a 9p mount).

The canonical repo stays on the WSL2 ext4 filesystem (fast iteration + the Linux
build); it is **copied** to `E:\DEV\rive-rust` only to run a Windows test (UNC
paths break MSBuild/MSVC, so building from `\\wsl.localhost\…` is not viable).

## 5. Gotchas encountered (Windows-specific)

1. **clang-cl is mandatory.** rive's sources are not MSVC-clean (e.g.
   `refcnt.hpp` deletes an incomplete `gpu::Texture` in GL-backend TUs that
   `cl.exe` rejects with C2027). rive's default toolset is `clang` → the
   **ClangCL** VS platform toolset; install "C++ Clang tools for Windows".
2. **premake beta7, not beta2.** beta2 renders rive's `fatalwarnings({'All'})` as
   the bogus MSVC flag `/weAll` (D8021). beta7 (rive's pin) emits `/WX`.
3. **`NOMINMAX`.** The Win32 loader path pulls in `<windows.h>`, whose `max`/`min`
   macros break `std::numeric_limits<uint32_t>::max()` in `vulkan_device.cpp`. The
   shim must define rive's Windows defines (`NOMINMAX`, `_USE_MATH_DEFINES`,
   `RIVE_WINDOWS`).
4. **CRT must match.** rive forces `/MT` (release static) on Windows; Rust must
   use `+crt-static` and `cc` must build the shim `/MT`, or the linker reports an
   `LNK2038` RuntimeLibrary mismatch.
5. **clang-cl shim ABI.** The shim is compiled with clang-cl (not `cl.exe`) so it
   sees rive's `__clang__` SIMD representation (`ext_vector_type`), matching the
   libs and avoiding a struct-layout mismatch across the FFI boundary.
6. **`link.exe` shadowing.** Git's `usr/bin` contains a coreutils `link.exe`;
   it must come **after** the MSVC paths on PATH (the relay appends it).
7. **vs dev shell defaults to x86.** Use `vcvars64.bat` (not the bare
   `Launch-VsDevShell`) for the x64 environment; MSBuild `-p:Platform=x64` builds
   x64 libs regardless.
8. **`.lib` naming.** On MSVC/clang-cl there is no `lib` file prefix and the
   double-prefix (`liblibpng.a`) disappears (`libpng.lib`). cargo link-names are
   identical on both OSes (`static=libpng` → `liblibpng.a` / `libpng.lib`).
9. **MSRV.** The workspace pins `rust-version = "1.94"`; the Windows box was 1.93
   (`rustup update stable` → 1.96).

## 7. Notes carried into M1a

- The ECS-facing API does **not** exist yet (M1a) and nothing here constrains it.
- `WGPU_BACKEND=vulkan` is set by the relay but has no effect yet (the shim
  self-manages its own `VkInstance`/`VkDevice`; no wgpu in the loop). It is set so
  M1a/M1b inherit it.
- Native Windows runs the **interlock** PLS path (vs WSL2's atomic Dozen path),
  which matters for M1b's zero-copy fast path; `out_win*.png` here is a trustworthy
  cross-platform reference for diffing M1a's CPU-copy result.
- The same Vulkan path now builds on Windows + Linux from one `build.rs`; the
  per-OS premake action and toolset are the only differences.
- **Hermeticity**: CI installs the Vulkan SDK (`.github/workflows/windows.yml`) so
  a clean Windows checkout generates SPIR-V itself — no Linux→Windows dependency.
  `build.rs` fails fast if neither the SDK nor prebuilt SPIR-V is present (§2).
- **Perf is not yet measurable**: rive is built debug — treat all M1.0/M1a
  timings as meaningless until M2 builds it optimized (`lld-link`, or
  `--config=release --no-lto`; §3b). Do not tackle this in M1a.
