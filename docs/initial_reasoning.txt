## The right mental model

Don't think of this as "make Rive draw into Bevy's scene." Think of it as two things:

1. **Keep rive-bevy's ECS surface and its offscreen-texture model unchanged.** rive-bevy already advances a state machine and renders Rive content into an offscreen Bevy `Image`, then displays that image via a material on sprites, UI nodes, or 3D quads (the `ui-on-cube` example). The Vello backend is just *what fills that texture*. Your job is to swap the fill, not the compositing. That keeps every downstream example/material working and makes this a backend swap rather than a rewrite.

2. **The actual engineering is GPU resource sharing + frame synchronization** between Rive's native `RenderContext` and Bevy's wgpu device. That's where reliability and performance are won or lost.

## The core decision: shared device vs external memory

For your target (single RTX 4090, desktop/Steam, Linux+Vulkan as the dev path), go **shared-device, Vulkan-first**. This is both the most performant and, counterintuitively, the most reliable, because you avoid cross-device memory import and the queue-ownership-transfer dance entirely.

The enabling fact: rive-runtime's Rive Renderer supports Metal, Vulkan, D3D12, D3D11, and OpenGL/WebGL, and exposes an abstract Renderer interface for hooking up an external vector renderer. And the "bring your own device" pattern already exists in the wild — RiveSharp's interop layer exposes a renderer-device-create-vulkan entry point that accepts host-provided VkInstance/Device/Queue handles, enabling bring-your-own-swapchain scenarios, with D3D12 HWND and CAMetalLayer interop already live end-to-end. That's a concrete reference C ABI you can mirror (more on that below).

The alternative — Rive renders on its own device and you import the result via shared handles (Vulkan `VK_KHR_external_memory`, D3D shared handle/keyed mutex, Metal `IOSurface`) — is real and wgpu supports pieces of it, but it's strictly worse here: it adds cross-device semaphore sync and forces image queue-ownership transfers through `VK_QUEUE_FAMILY_EXTERNAL`. Only reach for it if you ever need cross-process or a genuinely separate device.

## The concrete shared-device pipeline

**Let wgpu own the device, then lend it to Rive.** The historically painful part — creating the Vulkan device yourself so you can enable interop extensions — is now solved: wgpu_hal::vulkan::Adapter::open_with_callback lets you modify the pnext chains and extension lists wgpu populates before it creates the Vulkan device, which vastly simplifies interop versus having to create the device yourself. Use that to enable whatever the Rive Renderer needs (fragment-shader-interlock / rasterizer-ordered-access for its PLS-equivalent path, plus any external-semaphore/memory extensions you decide to keep).

Then:

- Extract `VkInstance`/`VkPhysicalDevice`/`VkDevice`/`VkQueue` from wgpu via the `as_hal` guard and hand them to Rive's `RenderContext` constructor (the RiveSharp-style FFI). Now both renderers live on the *same* logical device and queue.
- **Allocate the render-target texture in wgpu** (with `RENDER_ATTACHMENT | TEXTURE_BINDING` usage), then pull its `VkImage` out via `Texture::as_hal` and wrap that image as a Rive `RenderTarget`. One image, two API wrappers, zero copy. This is much cleaner than letting Rive allocate and importing into wgpu with `create_texture_from_hal` — note that create_texture_from_hal is treated as an internal wgpu API with no stability guarantees and needs a separate implementation per backend. By having wgpu allocate, Bevy already owns and tracks the texture; Rive just writes into it.

**Synchronization is the whole game.** I'd build it in three escalating tiers so you ship something correct fast and optimize later:

1. *v0 — fence/poll.* Submit Rive's frame, then `device.poll(Wait)` (or a fence) before Bevy samples. Trivially correct, one CPU stall per frame. Get the whole pipeline rendering correctly here before touching anything else.
2. *Steady state — same-queue ordering + a barrier.* Since Rive and wgpu share the queue, submission order gives you implicit GPU ordering; you just need to transition the image from Rive's color-attachment layout to Bevy's shader-read layout. wgpu now gives you exactly this hook: CommandEncoder::transition_resources() was added for native API interop, to let users insert barriers. No CPU stall.
3. *Async only if you need it — timeline semaphores.* If you ever split Rive onto a separate queue, drop to hal-level submit with signal/wait semaphores. Overkill for v0/v1.

**Drive all of this from a custom Bevy render-graph `Node`**, ordered before the camera pass that samples the texture. That's what makes the frame deterministic: advance state machine → Rive draw into the shared `VkImage` → barrier → Bevy samples. Don't try to make Rive draw directly into Bevy's main pass alongside 3D geometry; the PLS/depth interactions make that fragile. Offscreen-into-a-material is both Bevy-idiomatic and what rive-bevy already assumes.

The Rust side is where most of your time will actually go, and it splits cleanly:

The C++ build. rive-runtime builds with premake5, the dev team primarily works on macOS, and Windows/Linux support is community-contributed. You'll wrap that as a static lib and drive it from a `build.rs` (via the `cc`/`cmake` crate). Budget real time here — premake-on-Linux for the Rive Renderer is the least-trodden path and the most likely place to lose a day.

The C ABI. The Rive Renderer's C++ API (`RenderContext`, `RenderTarget`, the frame begin/flush calls) isn't directly FFI-able, so you need a thin C shim. You don't have to design it blind: the RiveSharp interop plan is the closest existing reference, exposing surface/swapchain create-resize-present-destroy entry points mirroring the renderer's own fiddle_context patterns for Metal, Vulkan, and D3D12. Mirror that shape. The `fiddle_context` files in the renderer's own test harness are the canonical example of how to bring up a `RenderContext` against a given device — read those first; they're effectively the reference embedding.

The Rust integration point. Rather than a greenfield crate, extend `rive-rs`: it already gates its current backend behind a `vello` feature, so add a `rive_renderer` feature implementing the same `Renderer` trait against your C shim. That keeps `rive-bevy`'s component layer (`RiveStateMachine`, the sprite/mesh entities) unchanged — you're swapping the trait impl, not the engine plugin. This is also exactly the direction Rive's own "underway" effort is taking, which means your work has a plausible upstream home.

## The sharp edges (where reliability actually lives)

The version triangle is your biggest ongoing tax. Bevy ships a pinned wgpu and bumps it with breaking changes roughly quarterly; `as_hal`/`create_texture_from_hal` are explicitly unstable internal APIs; and if you go Vulkan-raw you also pin an `ash` version that must match wgpu's. So you're locked to a (Bevy ↔ wgpu ↔ ash) triple and every Bevy upgrade is a potential interop re-validation. Pin hard, document the triple, and treat upgrades as a deliberate task, not a `cargo update`.

Device feature negotiation. The Rive Renderer's high-quality path wants fragment-shader-interlock / rasterizer-ordered access (its PLS-equivalent) and falls back to a slower atomic mode where that's unavailable. On a 4090 you'll have interlock, but you must enable the extension at device-creation time — which is precisely what `open_with_callback` is for. Get this wrong and you silently drop to the atomic path or fail device creation.

Format, usage, and color. The shared texture needs matching format and usage on both sides (`RENDER_ATTACHMENT | TEXTURE_BINDING` in wgpu, the equivalent `VkImageUsageFlags` for Rive). The classic bug here is color management: Rive renders premultiplied alpha and has opinions about sRGB; if Bevy's material/compositing assumes straight alpha or applies a second gamma, you get dark fringes or washed edges. Decide the colorspace contract once and assert it.

Ownership and lifetime. When you hand wgpu a raw image or hand Rive wgpu's handles, exactly one side must own destruction. wgpu's hal interop encodes this through drop-guards and `handle_is_owned` flags — and their behavior is inconsistent across `texture_from_raw` vs `device_from_raw`, which has bitten people. Be explicit about who frees the `VkImage` and the device, or you'll get use-after-free on shutdown or a leak.

## A strategic call before you write code

Because Rive themselves have the native backend "underway" for rive-bevy, you have a fork in the road: build your own interop now, or wait/track theirs. Given how this kind of work tends to go — and your pattern of building the substrate yourself — I'd prototype your own Vulkan shared-device path now (you learn the exact failure modes, you're unblocked), but structure it as a `rive_renderer` feature on `rive-rs` so it's upstreamable rather than a private fork you maintain forever. Worth a quick look at the rive-bevy/rive-rs issue trackers to see how far along their native effort is before you commit — that single data point could save you the C-ABI work entirely if they're close.

## Suggested de-risking order

The trap is trying to get shared-device-plus-zero-copy-plus-no-stall working all at once. Stage it so each step is independently verifiable:

First, prove the C++ build and C shim in isolation — render a `.riv` to an offscreen image via the Rive Renderer in a standalone Rust binary, no Bevy, no interop, dump to PNG. This validates the hardest, least-documented part (premake + FFI + RenderContext bring-up) before any wgpu complexity enters.

Second, prove device sharing with the dumb-but-correct sync — wgpu owns the device via `open_with_callback`, Rive renders into a wgpu-allocated `VkImage`, and you `poll(Wait)` before sampling. Display it on a Bevy sprite. If the image is correct, your entire resource-sharing model is validated; only performance remains.

Third, replace the stall with `transition_resources()` and same-queue ordering. This is a pure performance change over a known-correct baseline, which is exactly when such changes are safe to make.

Only after that does it make sense to think about Metal/D3D12 backends or async/multi-queue — and by then you'll have a working reference to port rather than three half-built paths.
